What is Testing?

Software systems are integral to our daily lives. Most people have encountered software that did not work as expected. When software malfunctions, it can lead to various problems such as financial losses, wasted time, damage to business reputation, and in severe cases, harm or even fatalities. Software testing is crucial for assessing software quality and reducing the risk of operational failures.

Testing involves multiple activities aimed at finding defects and evaluating the quality of software artifacts, known as test objects. It's a misconception that testing only involves executing tests (running software and checking results). Testing also includes planning, management, estimation, monitoring, and control, and it must align with the software development lifecycle.

Testing is often misunderstood as merely verifying compliance with requirements. While verification (ensuring the system meets stated requirements) is part of testing, validation (ensuring the system meets user and stakeholder needs) is equally important.

Testing can be dynamic (involving software execution) or static (not involving execution). Static testing includes reviews and static analysis, while dynamic testing uses various techniques to create test cases.

Effective testing requires intellectual effort, specialized knowledge, analytical skills, critical thinking, and systems thinking. Testers must use tools but must also think critically and analytically.

Test Objectives

Typical objectives of testing include:

  • Evaluating work products like requirements, designs, and code
  • Triggering failures to identify defects
  • Ensuring adequate coverage of the test object
  • Reducing risks associated with poor software quality
  • Confirming compliance with specified requirements
  • Ensuring the test object meets contractual, legal, and regulatory requirements
  • Providing stakeholders with information for decision-making
  • Building confidence in the test object's quality
  • Validating that the test object functions as expected by stakeholders

Testing objectives vary based on context, including the test object, test level, risks, software development lifecycle (SDLC), and business-related factors such as corporate structure and time to market.

Testing and Debugging

Testing and debugging are distinct activities in software development. Testing identifies defects through dynamic (executing software) or static (non-executing) methods.

In dynamic testing, failures triggered by defects are addressed through debugging, which involves:

  • Reproducing the failure
  • Diagnosing the root cause
  • Fixing the cause

After fixing, confirmation testing verifies the fix, ideally by the same tester. Regression testing ensures new fixes don't introduce new failures elsewhere.

Static testing directly identifies defects without software execution, requiring only defect removal, not failure reproduction or diagnosis.

Testing and Quality Assurance (QA)

Testing and quality assurance (QA) are often confused but are distinct. Testing is a form of quality control (QC), a product-oriented, corrective approach focused on achieving appropriate quality levels. Testing is a primary QC method, alongside formal methods, simulation, and prototyping.

QA, on the other hand, is a process-oriented, preventive approach focused on implementing and improving processes. The principle of QA is that a good process, when correctly followed, results in a good product. QA applies to both development and testing processes and requires adherence from all project participants.

Test results inform both QC and QA. In QC, results identify and fix product defects. In QA, results provide feedback on development and testing processes, aiding process improvements over time.

Errors, Defects, Failures, and Root Causes

Humans make mistakes, leading to defects (faults or bugs) that can cause failures. Mistakes can stem from various factors like time constraints, task complexity, process intricacies, infrastructure issues, fatigue, or insufficient training.

Defects can be present in documentation, source code, or supporting artifacts. Undetected defects in early SDLC stages can propagate, causing later-stage artifacts to be defective.

Executing defective code can cause failures, where the system behaves unexpectedly. Some defects always cause failures, others trigger failures in specific scenarios, and some may never cause failures.

Failures can also arise from environmental factors, such as radiation or electromagnetic interference affecting firmware.

Root cause analysis identifies the fundamental reasons behind problems (errors), aiming to prevent similar future issues by addressing these underlying causes, possibly removing them.

Principles of Testing

Testing principles provide general guidelines applicable to all testing. This syllabus describes seven such principles:

  1. Testing shows presence, not absence of defects: Testing reveals defects but cannot guarantee no defects. Even if no defects are found, it doesn't mean the test object is defect-free.
  2. Exhaustive testing is impossible: Comprehensive testing is infeasible except in simple cases. Focus on test techniques, prioritizing cases, and considering risks to guide testing efforts.
  3. Early testing saves time and money: Finding and fixing defects early prevents them from causing more issues later. Start static and dynamic testing early to catch problems sooner.
  4. Defects cluster together: Most defects are often found in a few system components, responsible for most failures (Pareto principle). Identifying defect clusters helps focus testing efforts.
  5. Tests wear out: Repeating the same tests can make them less effective at finding new defects. Modify tests or create new ones to maintain effectiveness, though repeated tests can be useful in automated regression testing.
  6. Testing is context dependent: Testing methods vary based on context, such as project, team, or technology used.
  7. Absence-of-defects fallacy: Believing that verifying software (meeting requirements and fixing defects) ensures system success is a fallacy. Even if requirements are met and defects fixed, the system might still not meet user needs, business goals, or be competitive. Validation checks if the system meets user needs.

Importance of Testing

Testing is a vital quality control measure that helps achieve objectives within scope, time, quality, and budget constraints. It is not solely the responsibility of the testing team; all stakeholders can leverage testing skills to contribute to project success. By thoroughly testing components, systems, and documentation, defects can be identified and addressed, thereby advancing the project towards its quality, functionality, and completion goals. Testing identifies software defects efficiently, allowing for their removal through debugging, which indirectly improves the quality of the test object. It directly evaluates the quality of the test object at various stages of the Software Development Life Cycle (SDLC), providing essential inputs for project management decisions, such as determining software release readiness. Testing indirectly ensures user needs are considered throughout the development process, which is crucial as direct user involvement is often impractical. It may also be required to comply with contractual, legal, or regulatory standards, ensuring the software meets legal and industry requirements. This compliance contributes to the overall quality and reliability of the software. By integrating these contributions, testing ensures that software projects are more likely to meet their intended goals. Ultimately, thorough testing delivers high-quality products that satisfy both stakeholders and users.
